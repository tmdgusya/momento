# Memento-Skills Plugin Design

Claude Code plugin implementing the Read-Write Reflective Learning loop from [arXiv:2603.18743](https://arxiv.org/abs/2603.18743). Enables Claude to autonomously create, track, and improve reusable skills from task execution experience.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Judge | Self-evaluation | Claude analyzes its own execution trace for success/failure |
| Skill scope | Global + project (user chooses) | Some patterns are universal, others project-specific |
| Architecture | Plugin + Hook (Approach B) | Hook enables automatic Read phase; skills handle Write phase |
| Skill promotion | None (single tier) | Self-judge already validates at creation; utility proves value over time |
| Auto-deletion | Never | Always requires user confirmation |

## 1. Metrics System

### Storage Locations

```
~/.claude/memento/metrics.json          # global
{project}/.claude/memento/metrics.json  # project-specific
```

### Schema

```json
{
  "version": 1,
  "config": {
    "utility_threshold": 0.4,
    "min_samples_for_judgment": 3,
    "prune_after_days_unused": 60
  },
  "skills": {
    "learned-api-retry-pattern": {
      "scope": "global",
      "created_at": "2026-03-25",
      "last_used": "2026-03-25",
      "last_optimized": null,
      "usage_count": 0,
      "success_count": 0,
      "failure_count": 0,
      "utility": null,
      "optimization_count": 0,
      "trigger_log": []
    }
  }
}
```

### Utility Calculation

```
U(skill) = success_count / (success_count + failure_count)
```

- `usage_count < min_samples_for_judgment` в†’ `utility = null` (judgment deferred)
- `U < utility_threshold` AND sufficient samples в†’ optimize candidate
- `last_used` older than `prune_after_days_unused` в†’ prune candidate

### trigger_log Entry

```json
{"task": "short task description", "result": "success|failure", "date": "2026-03-25"}
```

Capped at 5 most recent entries per skill.

## 2. Learned Skill Format

### Storage Locations

```
~/.claude/skills/learned-{name}/SKILL.md          # global
{project}/.claude/skills/learned-{name}/SKILL.md   # project-specific
```

`learned-` prefix distinguishes from manually authored skills.

### Template

```markdown
---
name: learned-{name}
description: Use when {trigger conditions extracted from self-evaluation}
---

# {Skill Name}

> Learned skill -- auto-generated by memento-skills
> Utility: {score} ({success}/{total}) | Last optimized: {date}

## When to Use
- {specific trigger situation}

## Procedure
1. {step}

## Known Failure Modes
- {failure mode}: {mitigation}

## Examples
{core pattern extracted from successful execution}
```

Constraints: max 500 words, description max 500 characters, CSO-optimized triggers.

### Lifecycle

```
create в†’ use в†’ judge в†’ optimize/prune
  в”‚               в”‚          в”‚
  в–ј               в–ј          в–ј
metrics init   utility update  threshold check
```

No draft/validated/reviewed tiers. Created directly into `learned-*/`. Utility proves value.

## 3. SessionStart Hook

### Behavior

1. Read global `~/.claude/memento/metrics.json`
2. Read project `{project}/.claude/memento/metrics.json` (if exists)
3. Merge and generate summary table
4. Inject into system prompt via `hookSpecificOutput.additionalContext`

### Injected Context Format

```
<memento-skills-context>
You have learned skills from past experience. Use them when relevant.
After completing significant tasks, consider using /memento reflect.

## Learned Skills ({N} global, {M} project)

| Skill | Utility | Uses | Scope | Last Used |
|-------|---------|------|-------|-----------|
| learned-x | 0.83 (10/12) | 12 | global | 2026-03-25 |

[warnings for low-utility skills]
</memento-skills-context>
```

### Context Budget

- 0 skills в†’ minimal message only
- 1-20 skills в†’ full table
- 21+ skills в†’ top 15 by utility + warning candidates + count of remainder
- Hard cap: ~300 tokens

### Implementation

Bash script following superpowers `session-start` pattern. Uses `jq` for JSON parsing with pure-bash fallback. Emits platform-appropriate JSON (`hookSpecificOutput` for Claude Code, `additional_context` for Cursor).

## 4. Skills

### 4-1. memento-reflect (Write Phase)

**Trigger:** After significant task completion or explicit `/memento reflect`.

**Protocol:**
1. Analyze current conversation execution trace
2. Self-evaluate: success/failure + reasoning
3. If learned skill was used в†’ update metrics (utility, trigger_log)
4. If reusable pattern found в†’ suggest creation (user confirms, then `memento create`)
5. If skill failed в†’ suggest optimization

**Self-evaluation criteria:**
- User accepted result without corrections?
- Completed in one pass or required retries?
- Errors encountered?
- Final result meets user request?

### 4-2. memento-create (Skill Discovery)

**Trigger:** reflect identifies new pattern, or user direct invocation.

**Protocol:**
1. Ask user: global or project scope?
2. Generate SKILL.md following learned skill template
3. Register initial entry in appropriate metrics.json (`utility: null`)
4. Show created skill to user for confirmation

### 4-3. memento-optimize (Skill Refinement)

**Trigger:** Utility below threshold, or user direct invocation.

**Protocol:**
1. Analyze trigger_log failure cases for target skill
2. Revise SKILL.md (Procedure and/or Known Failure Modes)
3. Update metrics: `last_optimized`, `optimization_count`
4. Show diff to user

### 4-4. memento-stats (Dashboard)

**Trigger:** User wants skill status overview.

**Output:** Formatted summary from metrics.json:
- Total skill counts (global/project)
- Utility distribution
- Recent usage frequency
- Optimize/prune candidates

### 4-5. memento-prune (Cleanup)

**Trigger:** After reviewing stats, or user direct invocation.

**Protocol:**
1. List prune candidates (low utility OR long unused)
2. User confirms each deletion
3. Remove SKILL.md + metrics.json entry
4. Never auto-deletes

### Skill Relationships

```
reflect --discovers--> create
   |                      |
   | detects failure       | initializes
   v                      v
optimize              metrics entry
   |
   | persistent low utility
   v
prune <--candidates-- stats
```
